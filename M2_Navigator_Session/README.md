# M2: Navigator Session

*Solve a programming problem with a partner using the pair-programming technique.*

**Target Skills: Accessible Communication, Problem Solving**

## Task

Collaborate with a partner for a 30-minute pair-programming session during which you solve a programming challenge. Use the “Driver” and “Navigator” roles (Driver writes the code and syntax details; Navigator guides the coder and focuses on the bigger picture) to facilitate your discussion and solution.

## Process

First, my partner and I communicated about our coding experience to find a problem we both felt comfortable solving. We decided to choose a coding exercise in Python. We planned the technical details of our virtual call, such as screen-sharing, in advance so that the pair-programming session could be focused and productive.

Next, we facilitated the pair-programming session. We summarized the problem and created a general plan, then started coding, using the Driver and Navigator roles to keep our session productive. We were able to solve a simple challenge in Python using this framework.

## Reflection

Before the pair-programming session, I predicted that Navigator would be the harder role. After completing the session, I found that it truly was more difficult for me because it requires clear explanations as well as abstract thinking about the full scope of the problem. While these are difficult skills to achieve consistently, they are invaluable for pair-programming and technical communication in general.

One aspect that was more challenging than expected was developing guiding instructions for the Driver instead of providing the code. One aspect that was easier than anticipated was collaborating within the assigned roles because I found that engaged conversation rather than mere direction helped solve the problem more efficiently.

Before the session, I predicted that I might fall into a micro-management pattern as Navigator. I found that I was tempted to provide detailed code rather than general instructions, but caught this behavior and focused on developing guiding questions to help my partner, which helped us think through the logic of the code clearly.

One strategy I planned in advance for noticing a mistake by the Driver is making a note of it and coming back to it later, giving the Driver time to complete their thinking before I dive in with a solution. In practice, this is a good strategy to avoid interrupting the workflow of the Driver, but it is important to find a balance and provide the feedback when it is still relevant in the code.

This session helped me practice the skills necessary for collaboration on code, a unique type of group work. I appreciated my partner’s dedication to planning ahead and using specific and respectful language to keep the session positive and focused.
